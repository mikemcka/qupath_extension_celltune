## Background — the CN algorithm (6 steps)

Today CellTune clusters cells by their **marker-intensity** vectors (`ScatterPlotView` + `CohortClusterModel`, Smile k-means). CN clustering is the same machinery applied to a **different feature**: instead of a cell's own marker intensities, each cell gets a vector describing the **cell-type composition of its local spatial neighborhood**, and those vectors are clustered.

(1) phenotyped cells with (x,y)+type → (2) each cell's k-nearest (or radius) neighbors → (3) per-cell composition vector = fraction of each cell type among neighbors → (4) k-means on the pooled composition vectors → (5) assign each cell its CN, read mean composition to name it → (6) map back / quantify.

**Key insight for reuse:** the composition matrix is just another `double[][]` fed to the *same* k-means + standardization code the codebase already uses. The only genuinely new logic is the k-NN window builder and the composition counter — and the k-NN index is an adaptation of code that already exists for distance measurements.

**Confirmed decisions (from user):**
- **Cell type** = `cell.getPathClass()` (the live classification). Unclassified/ignored cells are excluded as *types* but still receive a CN.
- **Storage** = a numeric `"CN"` measurement per cell (non-destructive; preserves the phenotype). No PathClass overwrite.
- **Color toggle** = a one-click button in the dialog that flips the viewer's cell coloring between the **classification** (phenotype) and the **CN** measurement. Non-destructive — uses QuPath's overlay measurement mapper, never touches `getPathClass()`.
- **Scope** = current open image first. Cohort/project-wide is a documented fast-follow, not in this pass.
- **Outputs** = include the CN × cell-type enrichment heatmap (paper Fig 4B) + per-image CN-frequency CSV.

NeighborhoodModel — pure logic, no QuPath types (primitive-array in/out so JUnit can drive it with synthetic clouds, mirroring `DistanceMeasurementsDialogTest`):
```java
public final class NeighborhoodModel {
    private NeighborhoodModel() {}

    // k nearest neighbours of ANY type, excluding self. out[i].length <= k (edges / n-1<k).
    // NaN-coord points → empty list. Adapts the STRtree pattern from DistanceMeasurementsDialog,
    // using JTS STRtree.kNearestNeighbour(Envelope, item, ItemDistance, k).
    public static int[][] kNearestNeighborIndices(double[] xs, double[] ys, int k);

    // All OTHER points within `radius` (euclidean, same units). STRtree expanded-envelope query
    // + exact dx*dx+dy*dy <= r*r filter, self-excluded.
    public static int[][] radiusNeighborIndices(double[] xs, double[] ys, double radius);

    // [n][nTypes] row-normalised fractions. typeId in [0,nTypes); -1 = ignored (excluded from
    // counts AND normalization base). includeCenter=true counts the centre cell (paper default).
    // Empty/all-ignored window → all-zero row (caller flags as CN = -1).
    public static double[][] compositionMatrix(int[][] neighbors, int[] typeId, int nTypes, boolean includeCenter);

    public record ClusterResult(int[] labels, double[][] centroids, int kEffective) {}

    // KMeans.fit on the (optionally standardized) composition matrix; kEff = min(k, nRows);
    // centroids recomputed as per-cluster means (mirror ScatterPlotView). NOTE: the reused
    // ScatterPlotView path sets no seed and Smile's KMeans seeds internally, so results are
    // NOT bit-reproducible across runs — the cluster test asserts blob recovery by purity, not
    // label ids. (If true determinism is later required, it is new work: a fixed-seed Smile
    // path, not inherited from the reuse target.)
    public static ClusterResult clusterCompositions(double[][] composition, int k);

    // [nCN][nTypes] mean composition fraction per CN — feeds the enrichment heatmap + naming.
    public static double[][] cnMeanComposition(int[] labels, double[][] composition, int nCN);
}
```
Defaults: kNN **k = 10**, `includeCenter = true`, radius default **50 µm**, CN count **k = 10** — these match Schürch et al. 2020 (Cell, STAR Methods "Neighborhood identification"): a window of the *10 nearest neighbors including the center cell* (Euclidean on X/Y), each window converted to a per-cell-type *frequency* vector, windows clustered with `MiniBatchKMeans(k=10)`.

Standardize-before-clustering is a UI toggle routed through `ScatterMath.standardizeColumns`, **default OFF**. The paper clusters the **raw frequency vectors** directly — it does NOT z-score before clustering — so OFF is the paper-faithful setting; z-scoring up-weights rare cell types (see `<risks_notes>`) and is offered only as an opt-in. (Note: the paper's "neighborhood enrichment score" of −3…+3 in Fig 4B is z-score-*by-column of the per-CN mean composition* — a **display** transform for the enrichment heatmap, NOT part of the clustering. That belongs in Task 4 via `IntensityHeatmap.zScoreByColumn`, and is independent of this clustering toggle.)

Implementation note: the paper used scikit-learn `MiniBatchKMeans` (a scale optimization for its 258k cells); reusing Smile's full `KMeans` on a single image is fine and slightly more accurate. Keep the CN-count default at 10 to match the paper.

private MeasurementMapper cnMapper; // cached toggle state

private void toggleCnColoring() {
    QuPathViewer viewer = qupath.getViewer();
    if (viewer == null) { Dialogs.showWarningNotification(...); return; }
    OverlayOptions opts = viewer.getOverlayOptions();
    if (cnMapper != null && opts.getMeasurementMapper() == cnMapper) {

        cnMapper = null;

        toggleBtn.setText("Color by: Classification");
    }
    viewer.repaintEntireImage();
}
