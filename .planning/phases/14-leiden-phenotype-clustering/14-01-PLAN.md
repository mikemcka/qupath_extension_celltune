## Background — why Leiden, and the recipe

Today the scatter-plot dialog clusters cells **only** with k-means: it z-scores the selected-marker matrix of the active cells and calls `KMeans.fit(active, k)` (`ScatterPlotView.recompute()`, ~L692). k-means forces a fixed `k`, assumes spherical/equal-size clusters, and under-resolves rare immune populations.

Every reference tool (scanpy, scimap, SPACEc; PhenoGraph originally) uses the **same graph recipe** instead:
1. **kNN graph** in feature space (the z-scored marker matrix).
2. **Edge weighting** — Jaccard / shared-nearest-neighbour.
3. **Leiden** community detection with a **resolution** parameter (higher → more clusters; there is no `k`).

Leiden slots into the *same* insertion point as k-means; the only genuinely new logic is the feature-space kNN graph builder, the edge weighting, the Leiden call, and — for cohort scope — a kNN label-transfer assigner. Full research and sources: `.planning/notes/leiden-clustering-design.md`.

**Confirmed decisions (from user):**
- **Scope** = current-image **and** whole-project (cohort) in this pass (parity with k-means).
- **Cohort assignment** = **kNN label transfer** (majority vote of nearest sampled cells), mirroring scanpy `sc.tl.ingest`. NOT synthetic centroids — no reference tool uses centroids for Leiden, and averaging a non-spherical community into one point defeats the method. (Per-cluster mean marker profiles are still computed **for display only** — the assignment-pane heatmap.)
- **Controls** = a `Method {k-means, Leiden}` selector; when Leiden is chosen the "Clusters (k)" spinner is replaced by a **resolution** spinner plus a **reproducibility** toggle (seed + random-starts, mirroring the shipped k-means multi-restart checkbox). kNN graph-neighbours and edge-weighting stay sensible hidden defaults.
- **Library** = `nl.cwts:networkanalysis` (CWTS Leiden — same authors as leidenalg; the reference implementation the Python tools trace back to). Smile 3.1.1 has no graph/community-detection support.

<interfaces>
LeidenModel — pure logic, no QuPath types (primitive-array in/out so JUnit can drive it with synthetic clouds, mirroring `NeighborhoodModel`):
```java
public final class LeidenModel {
    private LeidenModel() {}

    /** Community labels 0..nClusters-1 for every input row, and the cluster count. */
    public record LeidenResult(int[] labels, int nClusters) {}

    // Feature-space kNN indices (Euclidean over d columns), self-excluded, out[i].length <= k.
    // NEW: NeighborhoodModel's kNN is spatial (x,y) only. Brute-force is acceptable for the
    // sampled pool (<= ~50k rows); revisit with an ANN index only if profiling demands it.
    public static int[][] featureKnn(double[][] rows, int k);

    // Build a symmetric kNN graph on `rows`, weight edges by Jaccard/shared-nearest-neighbour,
    // run the CWTS Leiden algorithm at `resolution` (with `randomStarts` restarts, `seed` for
    // reproducibility), and return per-row community labels. k = graph neighbours (default ~15-30).
    public static LeidenResult cluster(double[][] rows, int k, double resolution, int randomStarts, long seed);

    // kNN label transfer (scanpy sc.tl.ingest style): assign each query row the majority Leiden
    // label among its `k` nearest reference rows (ties -> lowest label). Builds one index over the
    // labelled reference (the pooled, z-scored, Leiden-labelled sample). For cohort assignment.
    public static int[] transferLabels(double[][] query, double[][] reference, int[] refLabels, int k, int nClusters);
}
```
Defaults: graph kNN **k = 15** (scanpy `n_neighbors`; PhenoGraph uses ~30 — expose only if needed), edge weighting **Jaccard/SNN**, **resolution = 1.0** (range ~0.1-3.0), **randomStarts = 10** when the reproducibility toggle is on (else 1), quality function **CPM or Modularity** (whichever the bundled CWTS API exposes cleanly; Modularity is the safe default). The Leiden label array flows through the *identical* downstream path as `km.y`; per-cluster mean marker profiles are recomputed post-hoc for the assignment-pane heatmap exactly as the k-means path does.

CWTS API note: the `networkanalysis` README documents the CLI; the programmatic classes (`Network`, `Clustering`, `LeidenAlgorithm`) are public but their exact constructor/method signatures MUST be `javap`-verified against the resolved jar before implementing (see Task 1) — do not discover mid-implementation.

<risks_notes>
- **CWTS API is CLI-documented.** The programmatic `Network`/`LeidenAlgorithm`/`Clustering` classes are public but signatures must be javap-verified (Task 1) — this is the single biggest unknown, gated first on purpose (mirrors the JTS-kNN gate in phase 13).

- **Performance.** Brute-force feature kNN is O(n²·d)/O(n·k·d); fine for the bounded sample and typical single-image active sets. If profiling shows a bottleneck, an approximate-NN dependency (HNSW) is the follow-up, not v1.
- **Standardisation reuse.** Leiden receives the identical `active[m][selCols]` z-scored matrix as k-means — no method-specific preprocessing, so feature selection / within-class gating / marker-subset re-standardisation all apply unchanged.
</risks_notes>
